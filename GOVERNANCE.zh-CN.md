# 治理

[English](GOVERNANCE.md)

Home Edge Bootstrap 采用维护者主导、证据驱动的治理方式。目标是让产品、适配器、兼容性、
发布与安全决策可审查，同时避免用户受到不受支持的网络变更或过度声明影响。

## 角色与权限

- **贡献者**：按贡献政策提出 issue、文档、fixture、适配器与代码。
- **审查者**：提供技术、文档、兼容性、许可或安全审查。审查本身不授予合并、成熟度、
  支持或发布权限。
- **适配器维护者**：在项目维护者授予的权限范围内，对指定适配器及其证据承担持续责任。
- **项目维护者**：负责仓库政策、批准与合并变更、协调安全报告、确定适配器成熟度并授权发布。

当前项目维护者为 `@yiheng8023`。新增维护者应具有持续可信贡献和已经证明的审查判断能力。

## 决策标准

常规变更通过 issue 与 pull request 审查决定。对于削弱安全、可移植性、许可清晰度、恢复、
可维护性或公开声明准确性的变更，维护者可以要求补充证据或拒绝接纳。

重要行为、适配器、兼容性或治理变更应记录：

1. 用户可见问题与受影响主体；
2. 能力与权限边界；
3. 已考虑的替代方案，包括复用成熟既有组件；
4. 合成证据与平台证据；
5. 回滚、失败与维护行为；
6. 适用时两种操作语言中的文档与支持影响。

无法达成共识时，由项目维护者作出仓库最终决定并记录理由。安全敏感细节可以在适合披露前
保持机密。

## 适配器生命周期

适配器成熟度与目标支持分类彼此独立：

- **外部集成**：在本仓库外维护，不形成项目成熟度或支持声明。
- **实验性适配器**：在具备维护者和合成证据后，为边界明确的开发而接纳；接口可能变化，
  不隐含现场支持。
- **社区维护适配器**：具备可问责维护、已记录契约、fixture、恢复行为和经过审查的支持边界。
- **已验证适配器**：在上述基础上，通过与其声明兼容范围相匹配的项目证据门禁，并保持
  可问责维护。

当前 Merlin 实现是已实现参考适配器；这一角色不等于取得“已验证”成熟阶段。

接纳或晋级需要：边界明确的适配器身份、能力契约、目标支持政策、fail-closed 检查、备份与
回滚行为、不含敏感信息的诊断、双语操作说明、许可清晰度和明确维护责任人。与既有适配器相似、
受欢迎或在单个目标成功都不能替代证据。当证据或维护能力不足时，实现可以继续保持外部状态
或较低成熟阶段。

当证据过期、维护责任消失、兼容性漂移或继续收录会误导用户时，项目维护者可以降级、冻结或
退役适配器；在实际可行时，决定必须保留迁移或回滚路径。

## 兼容性与声明

fixture 证明的是受控输入下的脚本行为，不会认证具体硬件、固件、服务商、网络或永久可用性。
每个适配器独立声明目标支持分类。公开说明、仓库 topic、issue label、发布说明与推广材料必须
使用同一套批准术语，且不得扩大证据范围。

## 发布

只有适用的源码、许可、多平台、制品和发布就绪门禁通过后，项目维护者才授权发布。pull request
已经合并、收到赞助或适配器成熟度调整，本身都不构成发布承诺。

## 赞助与影响

赞助不会购买合并权限、适配器成熟度、兼容性声明、支持优先级、发布时间、安全决策或功能权利。
重要利益冲突应向审查者和维护者披露。

## 治理变更

治理变更采用与其他重要变更相同的审查流程，需要明确问题、影响分析、验证或审查面，以及
维护者批准。
