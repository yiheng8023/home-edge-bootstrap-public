# 架构说明

> 中文版是英文 `docs/ARCHITECTURE.md` 的阅读映射；英文文件是项目真源。

本项目不是某一个路由器型号的专用笔记，而是一套按能力驱动、可迁移的家庭网络边缘代理生命周期协调器，不是只执行一次的初始化脚本；它能在重启或漂移后从观测状态重新进入流程。核心策略应保存在 Mihomo/Clash 兼容配置和脚本中；固件相关内容应收敛到薄适配器。首个已经实现的参考适配器面向官方 Asuswrt-Merlin 支持的华硕路由器，但不把项目绑定到某一个具体型号、固件族、服务商、运行时、UI 模板、controller 端口或某一套代理组命名。

## 分层

| 层级 | 文件 | 可迁移性 | 说明 |
|---|---|---|---|
| 策略 | `config/policy.env`, `config/mihomo-overlay.yaml` | 高 | 默认地区中立；容忍不同服务商命名，并支持可选的现场候选与地区约束 |
| 运行脚本 | `scripts/self-heal.sh`, `scripts/update-sub.sh` | 高 | BusyBox/POSIX shell，避免桌面系统假设 |
| 适配器 | `adapters/merlin/` | 中 | Asuswrt-Merlin 路径、cron、`/jffs`、ShellCrash 约定 |
| 离线载荷 | `bundle/` | 中 | 内核和插件二进制与架构、平台相关 |
| 物理前置 | 无 | 手工 | 刷固件、开 SSH、首次凭据仍是人工边界 |

### Merlin 生命周期所有权

| 类别 | 表面 | 升级 / 回滚 | 项目退出 |
|---|---|---|---|
| 可替换 kit | `/jffs/home-edge-bootstrap` 与经过校验的项目事务变体 | 原子替换；可恢复上一版 kit | 删除 |
| 项目辅助脚本 | `/jffs/scripts` 下固定 `home-edge-*` 辅助脚本白名单 | 替换或从恢复 kit 重新应用 | 只删除白名单 |
| 共享注册 | 精确受管 `services-start` 块和 cron 名 `home_edge_selfheal` | 对精确项目条目做协调 | 只删除精确条目 |
| 操作者状态 | `/jffs/home-edge-bootstrap-state` 下订阅与本地策略 | 保留 | 保留 |
| 恢复状态 | 稳定根目录下订阅/运行时备份与生命周期证据 | 保留；仅通过显式恢复操作消费 | 保留并报告 |
| 可再生状态 | 稳定根目录下默认订阅缓存 | 可再生 | 删除 |
| 外部运行时 | `/jffs/ShellCrash`、Mihomo 内核与运行时自有配置 | 只由另行授权的运行时操作触碰 | 不触碰 |
| 固件/无关状态 | Asuswrt-Merlin、无关脚本、任务、服务和用户文件 | 项目不拥有 | 不触碰 |

固定删除边界写在经过审查的适配器代码中。来源证据可以佐证所有权，但目标元数据不能提供任意
待删除路径。

运行时存在性、controller 认证、Dashboard 配置、Dashboard 可达性、订阅凭据/缓存、live 订阅消费、路线验证、调度器注册、启动钩子注册和终端拓扑是相互独立的证据维度。任何一个维度健康都不能自动升级另一个维度。

## 敏感出口保障

敏感出口保障与日常可用性、自愈恢复属于不同能力类别。它的核心不变量是**敏感出口身份连续性**：在有明确时限的敏感操作窗口内，可观察到的路径所有者与有效叶节点必须保持一致；一旦发生变化，必须先使该窗口失效，再允许操作继续。可达、低延迟、地区标签或顶层选择组健康，都不能证明这一不变量。

未来的保障实现必须把出口身份建模为证据元组，而不是绑定服务商或节点名称。该元组至少需要包含运行时权威层、有效叶节点、出口地址族与观测出口、网络/ASN 分类、DNS 路径、策略代次和观测时间。它必须使用有时限的租约，在租约内抑制自动切换，检测漂移，并只对敏感流程执行失败关闭，不能因此中断无关的普通连接。租约也不能授权或代替账号、支付、注册、金融或地区验证操作。

当前自愈链路优化的是可用性与受限恢复，并不是敏感出口保障实现，现有路线健康证据也不得被表述为敏感出口保障。本版本只定义问题模型与负面契约；未来实现仍需独立夹具、隐私审查、操作者控制、过期行为和端到端拓扑证据。

## 适配器契约

以下契约用于未来适配器准入，也用于当前参考实现逐步完成隔离。框架应负责通用控制流，适配器负责目标特定集成。某个目标能够运行相似代理核心，并不意味着对应适配器自动取得支持声明。每个适配器都必须：

- 检测能力并报告不确定性，而不是只按产品名称判断；
- 在任何 apply 路径前提供只读诊断与 plan 行为；
- 写入必须明确确认，失败时保留最近的安全状态；
- 备份受管状态，并提供可执行回滚路径；
- 输出健康、来源和恢复证据，同时不泄露凭据；
- 把目标特定路径、服务管理、防火墙、DNS 和运行时约定收敛在适配器边界内；
- 提供版本化、幂等的迁移路径，保留存在差异的操作者状态，并在冲突时失败关闭；
- 提供先计划后应用的项目退出路径，使用固定所有权边界并报告保留状态；
- 提供离线夹具、明确兼容范围和具名维护责任人。

未来设备、固件和运行时工作只能通过能力、安全、恢复、证据、维护、迁移和项目退出契约接入。
未实现的适配器不是受支持目标，也不得在操作者说明中写成已支持。

框架复用成熟代理运行时与设备管理界面，不要求适配器重新实现代理核心、固件仪表盘或客户端 UI。

自动修复仅限项目自有文件、受管钩子块、cron 注册、缓存和受限选择器策略。运行时重启、Dashboard 安装、订阅信任/导入、防火墙/DNS、软路由配置和终端变更仍由适配器或人工把关，除非未来适配器证明了安全且可回滚的事务。

当前 Merlin 实现早于完整适配器隔离。`config/policy.env`、`scripts/self-heal.sh` 与 `scripts/update-sub.sh` 等通用文件仍保留 Merlin/JFFS/ShellCrash 默认值。这些默认值属于当前参考实现表面，不是可移植框架契约；其他适配器不得隐式继承。第二个适配器接入通用 apply 路径前，必须把这些默认值迁移到适配器提供的配置边界之后。

当前 Merlin 适配器会在 `services-start` 中安装带标记边界的受管块。该块调用项目自有 reconciler，恢复恰好一个 `home_edge_selfheal` cron 注册，并保留 `HEAL_CRON_DRY_RUN` 策略。它不会重启运行时、导入订阅、安装 Dashboard，也不会修改防火墙/DNS。未来软路由工作是独立的适配器准入事项，不是这项 Merlin 事务的延伸。

Merlin 生命周期契约固定使用 `/jffs/scripts`。离线夹具即使采用替代 JFFS 根目录，`BOOTSTRAP_SCRIPT_DIR` 仍必须等于 `BOOTSTRAP_JFFS_DIR/scripts`。早于 reconciler 的部署属于升级对象，必须重新运行适配器；注册修复辅助脚本仅用于 reconciler 已部署、但钩子或 cron 已漂移的情形。

## 适配器生命周期

适配器成熟阶段与目标支持分类是两个不同维度。成熟阶段描述适配器实现在仓库中的治理与证据状态；`supported`、`unknown`、`unsupported` 等目标支持分类，则描述该适配器对某一个目标的判断。当前 Merlin 实现承担项目参考实现角色；在正式准入记录和更广泛发布/证据门仍不完整时，不声明其已取得本生命周期中的“已验证”成熟阶段。

| 阶段 | 含义 | 仓库姿态 |
|---|---|---|
| 外部集成 | 相关项目应用框架契约或受其启发 | 链接或记录互操作方式；不产生仓库内支持声明 |
| 实验性适配器 | 已有边界明确的实现，但证据或维护责任尚不完整 | 独立命名空间、明确实验标签、无默认 apply 路径 |
| 社区适配器 | 已有维护者、夹具、文档和声明的兼容范围 | 社区维护；支持声明严格限于已有证据 |
| 已验证适配器 | 在声明范围内通过完整安全、回滚、隐私、可移植性和发布门 | 可进入一等引导流程与发布支持 |

如果许可证、维护方式或发布节奏不同，保持外部集成可能比合并更安全。是否接纳取决于证据与可持续维护责任，而不是只看流行度或实现原理是否相似。

## 运行原则

路由器代理配置完成前，不能依赖代理专用下载。恢复可用边缘代理所需的内容，必须满足以下条件之一：

- 已在 `bundle/` 中；
- 路由器上已经存在；
- 可在本地网络或直连网络中获取。

GitHub 在中国大陆某些网络下可以直连访问，但速度和稳定性受网络环境影响，也可能阶段性中断。因此，GitHub 托管的发布资产可以作为可选准备来源，但不能成为代理建立前主流程的必需输入。本地 `check-no-wall-readiness` 脚本只检查已经存在的内容，不访问网络。

## 目标流程

1. 人工完成必要前置：固件、局域网访问、SSH。
2. 将本工具包复制到路由器。
3. 先以 plan 模式运行适配器。
4. 确认计划后再以 apply 模式运行。
5. 验证已配置的探测目标和至少一个与操作者实际用途相关的外部目标。
6. 接受 DRY-RUN 证据后，启用并验证自愈调度器及其重启恢复钩子。订阅刷新仍是显式人工动作；当前实现没有为它安装定时任务。

## 运行时发现模型

服务商配置可能呈现类似这样的组链路：

```text
主选择组 -> 可选地区或策略组 -> 可达具体路线
```

具体标签不是项目契约。脚本会自动发现 Mihomo controller，并按角色/名称启发式推断主可选代理组；如果某个配置结构比较特殊，可通过 `HEAL_GROUP`、`HEAL_GROUP_MATCH_REGEX` 和 `CLASH_API` 显式覆盖。
